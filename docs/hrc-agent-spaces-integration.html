hrc-runtime / agent-spaces integration map

HRC owns host session state, runtime lifecycle, launch artifacts, callbacks, and normalized event streams. Agent Spaces owns placement resolution, bundle materialization, harness adapters, SDK session execution, and the public host-facing client used by HRC.

Boundary Map

The dependency direction is intentionally one-way. HRC consumes packages published from ../agent-spaces; ASP does not import HRC packages.

hrc-runtime

Host/controller layer. Owns daemon HTTP contracts, sessions, runs, launch artifacts, tmux/headless orchestration, callbacks, SQLite state, and user-facing CLIs.

  • hrc-core: HRC DTOs and persistent state shapes.
  • hrc-server: adapter bridge and lifecycle executor.
  • hrc-cli, hrcchat-cli: target resolution and operator UX.
  • hrc-frame-render: lifecycle events to render frames.
HRC imports ASP published packages via Verdaccio

agent-spaces

ASP layer. Owns scope syntax, config-time placement, materialization, harness registries, SDK session execution, and the public host-facing AgentSpacesClient.

  • agent-scope: canonical agent/project/task/lane identifiers.
  • spaces-config: placement and bundle resolution.
  • spaces-execution: runtime planning, brain/tools env prep.
  • agent-spaces: public bridge client used by HRC.
Boundary invariant HRC source may depend on ASP packages by package name. ASP source must not import hrc-*, acp-*, gateway-*, or other downstream repos. A source scan found no hrc-* imports in ../agent-spaces/packages.

Data Flows

The same HrcRuntimeIntent moves through several execution lanes. The important distinction is whether Agent Spaces returns a process to launch or runs an SDK turn in-process and emits AgentEvent callbacks.

Target resolution
1 Operator target

hrc and hrcchat accept handles like an agent, project, task, or lane.

2 agent-scope

resolveQualifiedScopeInput canonicalizes to scope and lane refs.

3 spaces-config

Finds agents root, project root, target config, profile, harness, and bundle ref.

4 HrcRuntimeIntent

HRC embeds RuntimePlacement, harness intent, execution mode, prompt, and env policy.

5 Server request

Intent reaches ensureRuntime, dispatchTurn, or semantic handoff.

Interactive / tmux
1 HRC runtime row

Server creates or reuses a tmux runtime and normalizes placement correlation.

2 buildCliInvocation

HRC calls AgentSpacesClient.buildProcessInvocationSpec.

3 ASP materialization

Agent Spaces resolves placement, plans harness runtime, materializes spaces, prepares brain/tools env.

4 Launch artifact

HRC writes argv, env, cwd, callback socket, spool dir, correlation env, and optional Codex descriptor.

5 Hooks / callbacks

Wrapper posts launch events, continuations, hook-derived events, and completion back to HRC.

SDK turn
1 SDK runtime

HRC creates a transient transport: 'sdk' runtime and run row.

2 runSdkTurn

HRC chooses agent-sdk or pi-sdk from the provider.

3 runTurnNonInteractive

Agent Spaces creates the SDK session, runs the prompt, and maps unified session events.

4 AgentEvent

Callbacks become sdk.* HRC events and message buffers.

5 Continuation

HRC persists provider continuation and harness session JSON on success.

In-flight input
1 Active run contribution

HRC validates runtime, run id, fences, and contribution capability.

2 deliverSdkInflightInput

HRC calls queueInFlightInput on the shared Agent Spaces client.

3 ASP in-flight map

Agent Spaces finds the active SDK session by hostSessionId and runId.

4 Prompt / interrupt

Input is enqueued; interrupt_and_continue invokes session interrupt when available.

5 Ack event

HRC appends sdk.inflight_delivered with accepted status and pending count.

Headless Codex
1 Codex CLI spec

Agent Spaces may include codexAppServer in ProcessInvocationSpec.

2 Launch artifact

HRC stores the descriptor and OTEL auth metadata.

3 Direct harness import

launch/exec.ts calls runCodexAppServerOneShot from spaces-harness-codex.

4 Callbacks

Thread ids become OpenAI continuations; app-server events are posted to HRC callbacks.

5 Normalized lifecycle

HRC records headless turn completion, buffers, and lifecycle events.

Contract Details

Expand each contract for the system-level meaning and source of truth. These are the places where protocol consolidation should start.

Critical Placement and runtime intent

HRC responsibility

  • Accept HTTP and CLI input, then build or parse HrcRuntimeIntent.
  • Normalize correlation with hostSessionId, runId, scopeRef, and laneRef.
  • Persist the intent on the HRC session as lastAppliedIntentJson.
  • Choose execution lane: tmux, headless, or SDK.

ASP responsibility

  • Resolve RuntimePlacement to ResolvedRuntimeBundle.
  • Materialize spaces, prompts, hooks, MCP config, and harness-specific files.
  • Plan effective cwd, model, yolo flag, and provider-specific run options.
  • Return either a process invocation spec or SDK events.
Critical Invocation spec and launch artifact

BuildProcessInvocationSpecRequest carries placement, provider, frontend, model, interaction mode, I/O mode, continuation, prompt, attachments, and env deltas. Agent Spaces returns ProcessInvocationSpec with argv, cwd, env, display command, resolved bundle, warnings, optional system prompt file, and optional Codex app-server descriptor.

HRC then writes HrcLaunchArtifact with HRC-only metadata: launchId, runtimeId, generation, callback socket, spool dir, OTEL configuration, and extracted correlation env.

Runtime Event conversion and projection

SDK execution uses Agent Spaces AgentEvent callbacks. HRC maps event types to sdk.state, sdk.message, sdk.message_delta, sdk.tool_call, sdk.tool_result, and sdk.complete HRC events. HRC strips fields already represented by the event envelope while preserving continuation when present.

CLI execution uses hook callbacks, launch callbacks, and Codex OTEL/app-server events. hrc-frame-render then adapts HRC lifecycle events to render frames, while relying on spaces-runtime message/tool result primitive shapes.

Core Correlation env

There are two related correlation env builders. Agent Spaces exports buildCorrelationEnvVars, producing AGENT_SCOPE_REF, AGENT_LANE_REF, HRC_SESSION_REF, and AGENT_HOST_SESSION_ID. HRC's adapter additionally produces ASP_SCOPE_REF, ASP_AGENT_ID, ASP_PROJECT, ASP_TASK_ID, ASP_HANDLE, HRC_HOST_SESSION_ID, HRC_RUN_ID, and task context vars.

This should become a single versioned contract. Today, callers can observe different names depending on which path built the env.

Concerns

Areas where protocols, ownership, or implementation details leak across the HRC / ASP boundary. Severity is based on refactor risk and likelihood of drift.

Critical

Mixed legacy and placement request shapes

RunTurnNonInteractiveRequest and BuildProcessInvocationSpecRequest still require legacy aspHome, spec, and cwd fields even when placement is set and those fields are ignored. HRC passes placeholder values such as spec: { spaces: [] } and cwd: '/'.

Critical

RuntimePlacement is cast, persisted, and not fully validated by HRC

server-parsers.ts casts request JSON to import('spaces-config').RuntimePlacement. HRC later persists the intent and routes execution from it. Invalid placement can escape early validation until the ASP resolver or launch path rejects it.

High

Correlation env has two overlapping dialects

Agent Spaces and HRC each build correlation env vars. The overlap is useful, but the naming differs: AGENT_SCOPE_REF vs ASP_SCOPE_REF, AGENT_HOST_SESSION_ID vs HRC_HOST_SESSION_ID, plus HRC-only task vars. Tool authors may depend on path-specific names.

High

SDK paths prepare brain/tools env twice

HRC's buildSdkRequestEnv calls detectAgentLocalComponents, prepareAgentBrainRuntime, and prepareAgentToolRuntime, then Agent Spaces placement SDK execution prepares brain/tools again. Some operations are idempotent, but the split makes env ownership unclear.

High

HRC imports a harness-specific Codex subpath

launch/exec.ts imports spaces-harness-codex/codex-session directly to run runCodexAppServerOneShot. That bypasses the otherwise public agent-spaces client boundary and couples HRC to a harness package.

Medium

Target resolution is duplicated in HRC CLIs

hrcchat-cli/src/resolve-intent.ts explicitly mirrors hrc-cli resolution to avoid an hrcchat -> hrc-cli dependency. Both read profiles, merge project targets, infer provider, and build placement.

Medium

Deprecated cpSessionId remains in public ASP contracts

Agent Spaces request and event types still carry cpSessionId alongside hostSessionId. HRC strips or conditionally preserves it in event JSON. This is old control-plane vocabulary leaking through a modern host/session contract.

Medium

Untyped prompt extension on invocation result

HRC casts ProcessInvocationSpec to read optional prompts, systemPromptFile, and codexAppServer. systemPromptFile and codexAppServer are typed in the ASP spec; prompts is an HRC-side extension and should be formalized or removed.

Medium

dryRun defaults hide invalid placement during dispatch

HRC normalizes dispatch placement with dryRun: true when unset, and headless SDK start paths also default dry run. That supports tests and early startup flows, but it can delay real validation and create different behavior across interactive, headless, and SDK lanes.

Medium

Event primitive ownership is split

spaces-runtime defines unified session message/tool primitives, while hrc-events also defines hook-derived primitives. The shapes are close, but not a single contract. Renderers consume the common subset through hrc-frame-render.

Refactor Targets

Practical simplifications that reduce protocol surface area without changing the three-repo ownership model.

First

Split placement-only request types

Add placement-only AgentSpacesClient methods or overloads that do not require legacy aspHome/spec/cwd placeholders. Keep legacy SpaceSpec methods as explicit compatibility APIs until removed.

First

Publish one correlation env contract

Define a single source of truth, probably in agent-scope or spaces-config, for session/ref/task env names. HRC can append HRC-only launch identity vars after the shared block.

Next

Move target-to-intent resolution into ASP

Extract the duplicated hrc-cli and hrcchat-cli resolution logic into a shared ASP helper that returns RuntimePlacement, provider, frontend, and warnings. HRC CLIs should own UX, not profile merging.

Next

Validate placement at the HTTP edge

Export a runtime validator or JSON schema for RuntimePlacement from spaces-config. HRC should reject malformed placement before persisting it as lastAppliedIntentJson.

Runtime

De-duplicate brain/tools preparation

Pick one owner. Prefer Agent Spaces to prepare ASP runtime env for both CLI and SDK placement execution, while HRC supplies only HRC launch/task env and operator overrides.

Runtime

Hide Codex app-server behind the bridge

Move runCodexAppServerOneShot behind agent-spaces or spaces-execution so HRC consumes a harness-neutral headless execution contract instead of importing a harness package subpath.
