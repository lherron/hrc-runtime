T-04901 / Daedalus conditional approval incorporated

HRC JSON-RPC stdio control plane

Add a process-oriented JSON-RPC surface beside the launchd daemon, but keep a single state owner. The approved architecture is not an HTTP wrapper; it is a shared runtime-control kernel with HTTP and stdio adapters.

Approved with constraints State-owning mode first No implicit proxy fallback
Transport Choice rpc.initialize

HTTP Unix Socket

Existing launchd `hrc-server` path remains the default operator host.

JSON-RPC Stdio

Peer state-owning process over the same control service, usually with alternate roots.

Private Callback Socket

Runtime children still call back to a service-owned Unix socket.

Implicit Proxy Fallback

Rejected. Proxy mode, if present, is explicit and secondary.

Decision

Same API, different host.

The client-facing decision is simple: `hrc-sdk` gets a transport seam. The implementation decision is harder: HRC must first separate domain control from HTTP request handling.

Approved thesis

Implement JSON-RPC stdio as a peer transport over a single state-owning `HrcControlService`, while preserving the existing HTTP Unix-socket daemon as the default operator path.

Full parity is the default claim. If v1 maps less than the current public SDK, the catalog must mark omissions as preview out-of-scope. 

createHrcClient({ transport: "http", socketPath })

createHrcClient({
  transport: "jsonrpc-stdio",
  command: "hrc",
  args: ["rpc", "--stdio"]
})
Invariant

One state owner per root tuple.

For any `{ stateRoot, runtimeRoot, dbPath, tmuxSocketPath }`, only one HRC process may mutate state, own controllers, run reconcilers, fan out subscribers, or accept runtime child callbacks.

Allowed ownership

HrcControlService

DB, lock, tmux, ghostmux, broker controller, sweepers, subscribers.

HTTP adapter

Parses `Request`, calls DTO methods, formats JSON or NDJSON responses.

JSON-RPC adapter

Owns frames, ids, notifications, stdout purity, and process lifecycle.

Rejected ownership

Two state owners

Daemon plus stdio both mutating the same roots creates split brain.

HTTP-shaped service

Domain methods must not take `Request` or return `Response`.

Client callback channel

Runtime children must not depend on the external JSON-RPC connection.

Boundary

Kernel versus adapters.

Daedalus approved `HrcControlService` only if it means the runtime-control kernel. Transport mechanics belong outside it.

Service owns

  • DB handle and server lock
  • tmux and ghostmux managers
  • Harness broker controller
  • Startup replay and reconciliation
  • Zombie and active-run sweepers
  • Event and message subscribers
  • Launch callback ingestion
  • Runtime, session, run, and message domain methods

Service must not own

  • Bun `Request` or `Response`
  • JSON-RPC frame handling
  • stdout and stderr policy
  • Public socket discovery
  • HTTP route parsing and formatting
  • CLI presentation behavior
  • Implicit HTTP proxy behavior
Modes

Initialize must tell the truth.

`rpc.initialize` is the negotiation point. It must identify whether the process owns HRC state or is merely proxying to an existing daemon.

Mode State owner Lock behavior Launch callbacks Use
state-owning true Acquire lock. Fail clearly if owned. Private service Unix socket. Primary stdio architecture and parity evidence.
http-proxy false Does not acquire state lock. Proxied daemon behavior. Explicit secondary/internal compatibility only.
auto-fallback ambiguous Rejected after lock conflict. Misleading. Not allowed.
Registry

Expose by catalog, not by route table.

The current HTTP table mixes public API, compatibility endpoints, and internal launch/hook callbacks. The RPC catalog must classify each method and exclude internal callback endpoints from public JSON-RPC.

Public catalog

`rpc.*`, `hrc.*`, and `hrcchat.*` methods intended for clients.

Compatibility catalog

Legacy or bridge methods that remain mapped but are not the strategic surface.

Internal endpoints

`/v1/internal/...` launch and hook callbacks stay service-private.

Parity

Map the actual SDK inventory.

The initial method sketch is representative only. The real v1 catalog must be checked against the current `HrcClient` surface or explicitly marked preview.

Area RPC namespace Parity rule
Sessions, runtimes, runs, launches hrc.* Map or mark preview out-of-scope.
Attach, capture, turns, in-flight input hrc.* Preserve DTO and error equivalence.
Surfaces, bridges, sweeps, reconcile hrc.* Classify public versus compat before exposing.
Targets, messages, waits, watches, DM hrcchat.* Same catalog; no separate state owner.
App-sessions hrc.* Map current SDK behavior or declare preview omission.
Streams

Subscriptions, not hanging requests.

Event and message watches become explicit subscriptions over JSON-RPC notifications. The SDK keeps async-iterator ergonomics.

Event flow

  • `hrc.event.subscribe` returns `subscriptionId` and high-water mark.
  • `hrc.event` notifications carry event payloads.
  • `hrc.event.unsubscribe` closes the feed.

Message flow

  • `hrcchat.message.subscribe` replays matching durable messages.
  • `hrcchat.message` notifications carry new messages.
  • Cleanup runs on unsubscribe, cancel, close, and process exit.
Handshake

`rpc.initialize` is the contract seal.

Clients must be able to tell whether they are connected to a true state-owning HRC process or an explicit proxy.

Plan

Build the seam before the transport.

The work should move in phases that first prove existing HTTP behavior survives service extraction, then add stdio and SDK transport selection.

Contract and registry

  • Define RPC DTOs in `hrc-core`.
  • Add facet and SDK parity accounting.

Service extraction

  • Create `HrcControlService`.
  • Move parsing and formatting to adapters.

Stdio host

  • Add `hrc rpc --stdio`.
  • Bind private callback socket.

SDK seam

  • Add HTTP and JSON-RPC transports.
  • Keep compatibility constructor.

Explicit proxy

  • Optional secondary mode only.
  • Report `stateOwner:false`.

Parity and e2e

  • Install before claiming done.
  • Smoke real launch callbacks.
Validation

Evidence gates.

Daedalus required ten tests. The launch callback e2e is the highest-value smoke because it proves stdio mode is not merely a client facade.

Registry parity

Every public SDK method is mapped or marked preview out-of-scope.

Service extraction

HTTP adapters delegate to transport-neutral service methods.

Lock conflict

Default-root stdio fails clearly while launchd owns the lock.

Alternate-root stdio

Initialize, status, resolve/list, inspect/control, and shutdown work.

hrcchat parity

Targets, messages, waits, subscriptions, DM, and handoff match HTTP.

Subscriptions

High-water marks, replay, cleanup, cancellation, and backpressure.

Error mapping

Domain errors stay distinct from parse, protocol, and process failures.

Stdout purity

Only JSON-RPC frames on stdout under logs, warnings, malformed input.

Launch callback e2e

Installed `hrc` proves private callback socket and spool replay.

Proxy mode

If implemented, it is explicit, internal, non-owning, and not parity evidence.

Highest Risk

A shallow extraction could keep HTTP-shaped business logic and create immediate RPC drift. The first implementation milestone should prove that domain methods are DTO methods behind both adapters, with no `Request` or `Response` crossing into the control kernel.

The second major risk is a hidden proxy fallback after lock conflict. That would make tests pass against the daemon while failing to prove the state-owning stdio architecture.
