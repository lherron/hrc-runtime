HRC runtime architecture

Separate runtime intent from runtime control.

HRC already treats provisioning and runtime control as different decisions. The refactor should make that separation executable through route decisions, a valid-combination catalog, and controller-specific runtime state. Harness-broker now sharpens the target boundary: HRC should route and persist the decision, while broker-capable harness execution moves out of exec.ts.

Context

The boundary is real, but implicit.

Requests describe placement, provider, and desired harness family. Runtime records, run records, lifecycle events, and responses report HRC's selected runtime controller. Today those decisions are scattered through predicates, transport strings, and optional JSON fields.

Provisioning

Agent Spaces resolves bundle, cwd, prompt, env, model, and harness runtime details. Intent should not carry an HRC runtime-controller field.

Routing

HRC interprets the intent, decides reuse/create/adopt policy, and selects the controller, startup method, interaction mode, and turn delivery.

Runtime Control

Tmux, Ghostty, embedded SDK, harness-broker, and managed child-process controllers own different lifecycle, input, capture, and state semantics.

Harness Broker Direction

Brokered harnesses get their own controller.

The broker package is no longer just a possible launch helper. It owns invocation identity, process protocol, input policy, stop/dispose, continuation updates, permission negotiation, and normalized events. That makes it an HRC runtime-control boundary.

Current migration shape

Headless Codex still reaches through hrc-server/src/launch/exec.ts. The wrapper drives app-server one-shot behavior, parses output, posts callbacks, and carries harness-specific execution logic inside HRC.

Target control boundary

HRC should select runtimeController: 'harness-broker', ask Agent Spaces for a HarnessInvocationSpec, control the broker through its client protocol, and map broker events back into HRC runtime, run, session, and lifecycle state.

Contract effect

broker-invocation and broker-input become operations on the broker controller. managed-child-process remains for command runtimes and temporary direct harness wrappers, not as the long-term bucket for broker-capable harnesses.

Broad-strokes plan update

The markdown plan now adds harness-broker to RuntimeController, gives brokered Codex its own route-catalog row, adds broker invocation state to runtimeState, treats broker-input as turn delivery through invocation.input, and marks direct exec.ts harness launch as migration-only.

Decision: sdk and headless are not controllers.

SDK is expressed through harnessRuntime, startupMethod, or turnDelivery. Headless is expressed through interactionMode.

Decision: harness-broker is a controller.

The broker protocol has invocation identity, status, input, stop/dispose, continuation, permissions, capabilities, and normalized events. Those are runtime-control semantics, not just a child-process startup method.

Recommendations

Make the existing boundary executable.

The goal is not a large rewrite. The path is to name the hidden decision points, type the impossible states out of existence, and let server modules reflect the architecture already present in the contracts.

1. Promote harness-broker

Treat brokered harness execution as runtimeController: 'harness-broker', not as a hidden branch of managed child-process execution. HRC owns the broker client and persistence mapping; the broker owns harness process/protocol details.

2. Extract RuntimeRouteDecision

Centralize the current predicates into a pure routing function that returns model provider, harness family, harness runtime, runtime controller, interaction mode, startup method, turn delivery, and whether provisioning must normalize the harness to interactive.

3. Rename transport

Use runtimeController for runtime lifecycle/control semantics. Remove sdk and headless from controller values; model them as harnessRuntime, interactionMode, startupMethod, and turnDelivery.

4. Discriminate runtime state

Replace opaque optional tmuxJson and surfaceJson checks with runtime-controller-specific variants. Make it impossible to represent runtimeController: 'tmux' without tmux pane metadata or runtimeController: 'harness-broker' without broker invocation identity.

5. Rename Agent Spaces nouns

Move provider toward modelProvider, harness toward harnessFamily, and frontend toward harnessRuntime in Agent Spaces first, then mirror aliases in HRC.

6. Split server responsibilities

Extract small modules for routing, interactive runtime provisioning, SDK dispatch, broker control, managed child-process execution, launch artifact building, startup, turn delivery, and controller operations.

Catalog usage

  • Input validation is partial: reject impossible intent combinations early.
  • Route resolution searches the catalog for legal candidates, then applies policy.
  • Output validation is exact: every resolved route must match one catalog row and one startup/delivery pair.
  • managed-child-process headless Codex is migration-only once HRC consumes harness-broker directly.
Contract Sketch

Requests are partial; resolved outputs are exact.

Input contracts capture user preference and raw model selection. Output contracts record the exact route, exact model identifier, and controller-specific runtime state HRC chose. 

type ModelSelectionRequest = {
  model?: string | undefined
  reasoningEffort?: 'low' | 'medium' | 'high' | 'xhigh' | undefined
}

type AgentRuntimeIntent = {
  placement: RuntimePlacement
  modelProvider: ModelProvider
  harnessFamily?: HarnessFamily | undefined
  preferredHarnessRuntime?: HarnessRuntime | undefined
  interactionMode?: InteractionMode | undefined
  model?: ModelSelectionRequest | undefined
}

Request/input: partial preferences. No runtimeController, startupMethod, turnDelivery, or runtimeState. 

type ResolvedModelSelection = {
  modelProvider: ModelProvider
  requestedModel?: string | undefined
  modelId: string
  modelAlias?: string | undefined
  modelVersion?: string | undefined
  reasoningEffort?: 'low' | 'medium' | 'high' | 'xhigh' | undefined
}

Resolved output: modelId is authoritative. modelVersion is optional metadata for aliases or parsed provider IDs.
